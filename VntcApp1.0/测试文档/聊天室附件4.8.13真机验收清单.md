# 聊天室附件 v4.8.13 真机验收清单

## 验收环境

- 至少两台 Android arm64 设备，建议分别覆盖 Android 8～11 与 Android 12 以上系统。
- 两台设备安装或从 4.8.12 覆盖升级到 4.8.13。
- 两台设备加入同一 VNT 网络，记录是否同网、直连或中继。
- 测试前确认系统存在可打开 JPG、PNG 和测试文档的应用。

## 一、图片接收与打开

1. 设备 A 分别向设备 B 发送 JPG 和 PNG。
2. 等待接收完成，在设备 B 点击附件。
3. 确认系统图片应用正常打开内容。
4. 确认不再出现“打开附件失败”，也没有文件权限拒绝提示。
5. 从 4.8.12 覆盖升级的设备重复一次，确认升级路径正常。

通过标准：图片内容正确、可重复打开，应用不会崩溃，附件文件没有损坏。

## 二、私聊传输速度

1. 发送一个约 8 MB 文件和一个约 50 MB 文件。
2. 分别记录开始时间、完成时间、文件大小和界面显示速度。
3. 用“文件大小 ÷ 实际耗时”计算实际平均速度。
4. 比较界面速度与实际平均速度，并记录直连或中继状态。

通过标准：健康局域网/直连环境下不再长期停留在几十 KB/s；界面速度与实际上传吞吐同一量级，进度最终到达 100%。若仍慢，需同时记录 VNT 链路类型、延迟、丢包和服务端/对端 CPU，区分网络瓶颈。

## 三、多人聊天室

1. 三台或更多设备加入同一聊天室。
2. 一台设备发送 8 MB 以上附件。
3. 确认每个接收方都能完整接收并打开。
4. 观察发送端总进度和速度显示。

通过标准：总进度按所有接收方工作量递增并最终到 100%；速度不再因接收方数量被错误除小；不存在某一接收方成功但整体状态提前完成。

## 四、异常与回归

- 传输中暂时断网后恢复，确认续传或明确失败，不生成可误开的损坏文件。
- 接收端磁盘空间不足时应明确失败。
- 发送 PDF、TXT、ZIP 等非图片附件，确认交给兼容应用打开；没有兼容应用时显示明确提示。
- 验证文字消息、语音消息和原有分享/保存附件功能不受影响。

## 已完成的自动验收

- Android API 35 模拟器安装并启动正式 4.8.13 arm64 Release APK：通过。
- 两个独立 Android API 35 应用进程加入同一 VNT 网络并互相发现：通过。
- A 端发送文字，B 端接收并显示：通过。
- A 端发送真实 PNG，B 端接收后点击聊天气泡打开：通过。
- B 端将应用私有 PNG 转换为 FileProvider `content://` URI：通过。
- 前台 Activity 携带 `image/png` 和临时读取授权启动 Google Photos：通过。
- Google Photos `HostPhotoPagerActivity` 成为前台页面，无 FileProvider、权限拒绝或崩溃：通过。
- 8 MB 附件：A 端 2.76 秒完成，约 2.90 MB/s；B 端完整落盘：通过。
- 20 MB 通用 `.bin`：A 端 5.64 秒完成，约 3.54 MB/s；B 端完整落盘并显示 20.0MB：通过。
- 50 MB 附件：A 端 12.87 秒完成，约 3.89 MB/s；B 端完整落盘并显示 50.0MB：通过。
- 8 MB、20 MB、50 MB 传输均完成 `attachment_resume`、`attachment_ack`，发送结果均为 `attempted=1 delivered=1 failed=0 status=sent`。
- 对端不在在线成员列表时，附件显示发送失败；后续元数据同步在接收端明确显示“附件未自动补齐，需要发送方手动重发”，不会把缺失文件误标为可打开：通过。
- Android API 35 `dataSync` 前台服务启动：通过，`isForeground=true`。
- `FOREGROUND_SERVICE_DATA_SYNC` 权限异常复查：通过，无 SecurityException。

自动验收已覆盖两个独立应用进程之间的真实 VNT 聊天、附件传输、接收落盘、气泡点击和 Android 宿主权限链路。不同品牌真机、移动网络、覆盖升级和三人以上群聊仍建议按下列清单做发布现场回归。

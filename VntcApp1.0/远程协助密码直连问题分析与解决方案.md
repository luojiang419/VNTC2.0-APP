# 远程协助密码直连问题分析与解决方案

## 问题现象

- 其他用户在“远程协助”页面点击“设置远程密码”后，界面提示设置成功。
- 但我去连接他们时，仍然提示“等待对方接受”。
- 期望行为是：
  - 对方设置了非空远程密码后，我输入正确密码即可直接连接。
  - 对方清空密码后，恢复为必须本机手动点击接受。

## 结论先说

这不是单点问题，而是两个源码问题叠加导致的：

1. `vntcrustdesk` 的定制配置里，把 `approve-mode` 强制覆盖成了 `click`。
2. “设置远程密码”这条无界面命令，走的是 Flutter FFI 初始化链路，但桌面端这条链路没有执行 `vntc::apply_profile()`，导致它和真实后台服务的配置上下文不完全一致。

其中第 1 条是主因，而且是决定性主因。

只要 `approve-mode` 还被覆盖为 `click`，即使密码写进去了，服务端登录分支仍然会进入“等待本机接受”。

## 关键源码证据

### 1. 服务端确实会在 `approve-mode = click` 时强制走“等待接受”

文件：
- `D:\Myproject\vnt2.0\vntcrustdesk-src\src\server\connection.rs:2300`

关键逻辑：
- 当 `password::approve_mode() == ApproveMode::Click` 时，连接不会直接放行，而是进入 Connection Manager，等待本机接受。
- 这正好对应你现在看到的“等待对方接受”。

也就是说，只要被控端最终读取到的 `approve-mode` 是 `click`，密码直连就不可能成功。

### 2. `vntc` 定制配置把 `approve-mode` 强制写死成了 `click`

文件：
- `D:\Myproject\vnt2.0\vntcrustdesk-src\src\vntc.rs:24`

关键逻辑：
- `apply_profile()` 里向 `OVERWRITE_SETTINGS` 写入了：
  - `OPTION_APPROVE_MODE = click`

这不是默认值，而是覆盖值。

### 3. 配置系统读取顺序里，覆盖值优先级最高

文件：
- `D:\Myproject\vnt2.0\vntcrustdesk-src\libs\hbb_common\src\config.rs:1100`

关键逻辑：
- `Config::get_option()` 的优先级是：
  - `OVERWRITE_SETTINGS`
  - `CONFIG2.options`
  - `DEFAULT_SETTINGS`

这意味着：
- 只要 `OVERWRITE_SETTINGS` 里存在 `approve-mode=click`
- 那么用户 later 写入的 `password` 模式不会生效

### 4. 配置系统还会阻止保存被覆盖的选项

文件：
- `D:\Myproject\vnt2.0\vntcrustdesk-src\libs\hbb_common\src\config.rs:2476`
- `D:\Myproject\vnt2.0\vntcrustdesk-src\libs\hbb_common\src\config.rs:1114`

关键逻辑：
- `is_option_can_save()` 明确规定：
  - 如果某个 key 已经存在于 `OVERWRITE_SETTINGS`
  - 那么这个 key 不能作为用户配置保存

所以当前代码下，`approve-mode` 不只是“读出来总是 click”，而且“用户设置的 password 也根本存不进去”。

这已经足够解释为什么设置远程密码后，连接仍然要求手动接受。

## 第二个隐藏问题

### 5. Flutter 桌面初始化链路没有执行 `vntc::apply_profile()`

文件：
- `D:\Myproject\vnt2.0\vntcrustdesk-src\src\common.rs:124`
- `D:\Myproject\vnt2.0\vntcrustdesk-src\src\flutter_ffi.rs:40`
- `D:\Myproject\vnt2.0\vntcrustdesk-src\src\common.rs:1809`

关键结论：

- `common::global_init()` 会执行 `crate::vntc::apply_profile()`
- 但桌面 Flutter FFI 的 `initialize()` 并没有调用 `common::global_init()`
- 它只会尝试 `load_custom_client()`
- 而 `load_custom_client()` 在 `is_direct_only_mode()` 下会直接 `return`

这意味着：
- 后台服务/核心链路和 Flutter 无界面配置链路，可能不在同一套 `APP_NAME / OVERWRITE_SETTINGS / 配置上下文` 下工作
- 这会让“设置成功”的校验结果不可靠

### 6. 现有“设置成功”校验并不够可信

文件：
- `D:\Myproject\vnt2.0\vntcrustdesk-src\flutter\lib\main.dart:125`

当前逻辑：
- `--configure-access-password` 执行后
- 立即在同一进程里读取：
  - permanent password
  - verification-method
  - approve-mode
- 只要读回来的值匹配，就直接 `exit(0)`

问题在于：
- 这只能证明“当前这次命令执行上下文”里读到了这些值
- 不能证明：
  - 真正的后台服务已经切到 `password`
  - 配置写入到了被服务实际读取的那份配置
  - 受 `OVERWRITE_SETTINGS` 限制的 key 真正可持久化生效

## 为什么当前现象会稳定复现

按照当前代码，完整链路大致是：

1. 用户在主项目点击“设置远程密码”
2. 主项目调用：
   - `vntcrustdesk.exe --configure-access-password <password>`
3. 该命令尝试设置：
   - permanent password
   - `verification-method = use-permanent-password`
   - `approve-mode = password`
4. 但 `approve-mode` 在 `vntc::apply_profile()` 里被覆盖为 `click`
5. 配置系统读取时优先返回覆盖值 `click`
6. 服务端收到连接后命中：
   - `approve_mode() == Click`
7. 最终表现为：
   - “等待对方接受”

所以这不是偶发 bug，而是当前源码设计下的必然结果。

## 推荐修复方案

## 方案目标

- 默认仍保持“未设置密码时，需要手动接受”
- 一旦用户设置了非空远程密码，就允许密码直连
- 清空密码后，再恢复到手动接受

## 推荐方案 A

这是我建议执行的方案，改动最小，行为最符合你当前业务需求。

### A1. 不要再把 `approve-mode` 放在 `OVERWRITE_SETTINGS`

修改文件：
- `D:\Myproject\vnt2.0\vntcrustdesk-src\src\vntc.rs`

建议调整为：
- 保留这些强制覆盖项：
  - `direct-server`
  - `direct-access-port`
  - 禁止自动更新
  - 清空 rendezvous / relay / api / key
- 但把 `approve-mode = click` 从 `OVERWRITE_SETTINGS` 移除

更稳妥的做法：
- 把 `approve-mode = click` 改成写入 `DEFAULT_SETTINGS`
- 含义变成：
  - 默认是 click
  - 但允许用户后续改成 password

这样能保持初始安全策略，又不会把“密码直连”彻底锁死。

### A2. 在桌面 Flutter 初始化时补上 `vntc` profile

修改文件：
- `D:\Myproject\vnt2.0\vntcrustdesk-src\src\flutter_ffi.rs`

建议调整为：
- 桌面端 `initialize()` 时也执行 `crate::common::global_init()`
- 或至少显式执行 `crate::vntc::apply_profile()`

目的：
- 保证 `--configure-access-password` 这条命令
- 与真实后台服务
- 使用同一套应用名、配置路径和定制覆盖规则

如果不补这一步，即使修完 A1，也仍然可能出现：
- 命令写到一套配置
- 服务读另一套配置

### A3. 保留当前密码设置语义

当前语义本身是对的，不需要推翻：

- 非空密码：
  - `verification-method = use-permanent-password`
  - `approve-mode = password`
- 空密码：
  - `approve-mode = click`

这一部分只要 A1 生效，逻辑就能真正落地。

### A4. 增强 `--configure-access-password` 的校验

建议补充以下校验，而不是只读当前进程状态：

1. 读取 `Config::get_option("approve-mode")` 最终值
2. 读取 `Config::get_permanent_password()` 最终值
3. 如果后台服务存在，确认 IPC 更新成功
4. 必要时输出当前实际配置文件路径到日志

这样以后再遇到“界面提示成功，但实际没生效”时，定位会非常快。

## 不推荐的方案

### 不推荐方案 B：只在设置密码后重启服务

原因：
- 当前主因不是“服务没重启”
- 而是 `approve-mode` 本身被覆盖为 `click`

即使重启：
- 服务重新读取的仍然还是 `click`
- 问题依旧存在

所以单独重启服务不能解决根因。

### 不推荐方案 C：连接端强行绕过接受逻辑

原因：
- 服务端在 `connection.rs` 已明确以 `approve-mode` 决定授权路径
- 如果在连接端硬绕，会变成协议与授权模型分裂
- 风险高，后续更难维护

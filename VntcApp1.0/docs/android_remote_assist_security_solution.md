# Android 聊天、远程协助与安装安全解决方案

## 目标

本方案同时处理以下回归与安全问题：

1. 聊天室长期显示“消息监听未就绪”；
2. 系统已授权无障碍，但应用仍显示“待处理”；
3. 屏幕录制授权成功约 12 秒后自动回退为未开启；
4. RustDesk 受控端已启动，但 Android 端无法可靠判断直连监听器是否就绪；
5. 小米安装器把包含远控能力的正式 APK 识别为高风险木马；
6. 本地固定 Rust 工具链被 Cargokit 误判为未安装，造成构建无意义联网失败。

## 根因与最小修复

### 1. 聊天消息监听

- 根因：小米 Android 内核把 `50012-50029` 配置为本机保留端口，旧聊天端口 `50018/50019` 绑定时返回权限错误。
- 修复：聊天 TCP/UDP 监听端口迁移到 `61018/61019`，并同步 Windows 防火墙规则名与回归契约。
- 状态语义：只有两个监听器均真实绑定成功时才显示“消息监听已就绪”；失败时保留可读错误而不是静默显示待处理。

### 2. 无障碍授权与连接状态

- 根因：界面把“系统设置已授权”和“InputService 已连接”混成一个状态；输入通道停止时还会调用 `disableSelf()`，主动撤销系统授权。
- 修复：
  - 权限卡片以系统设置授权为准，显示“已授权/未开启”；
  - 实际远控能力继续以 InputService 活连接为准，显示“已连接/未连接”；
  - 停止输入或停止远控不再调用 `disableSelf()`；
  - 无障碍事件缩小到焦点、窗口状态和窗口集合变化，不再声明全部事件或上报 View ID。

### 3. 录屏授权延迟回退

- 根因：Dart 端等待 RustDesk `49999` 监听器 12 秒超时后调用 `stopService()`，把已经成功建立的 MediaProjection 一并销毁；首次录屏初始化还会无条件释放旧资源并广播未授权状态。
- 修复：
  - 监听等待超时只报告启动失败，不再回滚已经成功的录屏会话；
  - 首次初始化仅在确实存在旧投影/编码资源时释放；
  - 录屏“请求中、已授权、可用、失败原因”分开建模，避免瞬时状态覆盖真实授权。

### 4. RustDesk 直连监听状态

- 根因：Android SELinux 禁止应用读取 `/proc/self/net/tcp`，旧实现把读取失败误判为 `49999` 未监听。
- 修复：Rust 直连监听循环维护权威 `AtomicBool`，通过 JNI `isDirectServerListening()` 暴露给 Android；Java 不再依赖 procfs 推测。

### 5. 小米“木马/高风险”告警

- 结论：已审计 APK 与 GitHub Release 资产哈希一致，v2/v3 签名有效且证书为官方固定证书；未发现动态 Dex、未知控制域名、重签名器或第三方上传通道。高概率是 VPN、MediaProjection、无障碍手势、悬浮窗、相机/麦克风、开机服务和安装更新能力叠加触发厂商启发式分类。
- 已实施的安全收敛：
  - 更新 APK 必须位于应用私有更新目录；
  - 安装前和未知来源授权返回后都重新校验，防止 TOCTOU；
  - 必须匹配 `top.wherewego.vnt_app`、版本严格高于当前版本、且只有一个签名者；
  - 签名证书 SHA-256 必须匹配固定官方证书 `83EF92B147643697EA8EFAF4B73A9D370875EE0FD1CBA10A7AC03E184F44D8C7`；
  - 移除无意义的 quick settings uses-permission，并缩小无障碍事件范围；
  - Release 增加包名、版本、资产名、资产大小、源码提交、工作流运行、证书与 APK SHA-256 的签名证明 JSON。
- 边界：厂商病毒库属于外部分类系统，代码修改不能保证立即消除告警。若新版本仍被拦截，应向小米提交正式 Release URL、签名证明 JSON、APK/SHA256、固定证书指纹及本审计说明申请复核/加白。长期可把远控能力拆成用户主动安装的官方插件，降低主应用默认攻击面。

### 6. 固定 Rust 工具链识别

- 根因：Cargokit 只把 `stable/beta/nightly` 当作已安装工具链，忽略 `1.95.0-x86_64-pc-windows-msvc` 这类固定版本。
- 修复：非自定义工具链匹配增加 `x.y.z` 数字发布格式，避免每次构建都联网重装已存在工具链。

## 发布与供应链约束

- 本地 Android 原生构建固定 Rust `1.88.0`、cargo-ndk `3.1.2`、NDK `28.2.13676358`、FRB `1.80.1` 和 vcpkg 提交 `120deac3062162151622ca4860575a33844ba10b`。
- RustDesk ELF 必须通过仓库 16KB 页兼容脚本，所有 LOAD 段对齐至少为 `0x4000`。
- 本地测试 APK 和 GitHub 正式 APK 都必须使用同一官方固定证书；签名密钥不得进入仓库或构建日志。
- 正式 Release 必须为非草稿、非预发布，包含 APK、SHA-256 旁车和 Android 签名证明 JSON，并绑定同一 tag、源码提交和版本号。

## 验收矩阵

| 模块 | 必须通过的验收 |
| --- | --- |
| 聊天 | Android 上 TCP `61018`、UDP `61019` 真实监听；页面不再显示“消息监听未就绪”；双方文字消息可收发 |
| 无障碍 | 系统授权后权限卡显示“已授权”；停止/重启远控不撤销系统授权；服务重连后显示“已连接” |
| 录屏 | 授权后持续 60 秒以上仍为已开启；失败时不销毁已成功的 MediaProjection |
| 直连远控 | Rust JNI 报告 `49999` 真实监听；控制端可建立会话并至少持续 60 秒；触摸手势有执行回执 |
| 更新安装 | 错包名、旧版本、多签名或非官方证书 APK 均在启动系统安装器前失败关闭 |
| 本地 APK | 包名/版本正确、arm64 only、官方证书正确、RustDesk ELF 16KB 对齐、新 JNI 符号存在 |
| 云端发布 | Actions 全部成功；正式 Release 公开；资产、SHA-256、签名证明、tag 和提交一致 |
| 厂商告警 | 记录小米扫描结果；若仍告警，提交厂商复核材料，不把启发式告警消失作为代码正确性的唯一判断 |

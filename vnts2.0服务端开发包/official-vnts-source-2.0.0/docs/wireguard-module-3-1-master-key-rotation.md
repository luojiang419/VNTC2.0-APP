# WireGuard 模块 3.1：停机原子主密钥轮换

## 范围

本模块只轮换用于加密 SQLite 中 WireGuard 服务端身份私钥的 32 字节主密钥。X25519 身份私钥本身不变，因此服务端 WireGuard 公钥和未来客户端配置均保持不变。

本模块不实现在线双密钥、身份私钥轮换、peer 公钥、PSK、UDP、数据桥接、HTTP 密钥接口或客户端页面。

## 命令契约

```powershell
vnts2.exe `
  --conf C:\ProgramData\vnts2\config.toml `
  --rotate-wireguard-master-key C:\ProgramData\vnts2\wireguard-master-v2.key
```

- `--conf` 必须显式提供，轮换模式不会创建默认配置。
- 配置必须包含 `persistence = true` 和当前有效的 `wireguard_master_key_file`。
- 命令参数是新的主密钥文件；当前和新文件都必须是严格 32 字节二进制。
- 相对路径以配置文件所在工作目录为基准；生产环境建议使用绝对路径。
- 新主密钥内容必须与当前主密钥不同。
- 轮换模式不启动 TCP、QUIC、WebSocket、HTTP、peer server 或 Windows 服务入口。

## 停机互斥

正常持久化服务和轮换命令都会打开 `network_control.db.lock` 并持有标准库文件独占锁：

- 正常服务从 SQLite 初始化前持有，直到进程退出；
- 轮换命令必须取得同一锁；
- 另一个 VNTS 进程持锁时立即失败，不等待、不修改数据库；
- 进程退出或崩溃后操作系统释放锁，空锁文件可继续复用，不需要删除。

该锁用于 VNTS 进程间协作；直接绕过 VNTS 修改 SQLite 的外部程序不受保护，生产环境仍应限制数据库文件权限。

## 原子更新

轮换顺序：

1. 取得数据库进程独占锁。
2. 严格读取当前和新主密钥。
3. 使用当前主密钥认证解密现有身份，并重新派生/校验 X25519 公钥。
4. 将 `encryption_key_version` 加一并生成新的随机 24 字节 nonce。
5. 使用新主密钥和新版本 AAD 重新执行 XChaCha20-Poly1305 加密。
6. 执行单条 SQLite `UPDATE`，`WHERE` 同时匹配完整旧版本、旧 nonce、旧密文、公钥和时间戳。
7. 只有恰好更新一行才报告成功；记录已变化时拒绝覆盖。

密钥文件读取、旧密文认证、版本递增或新密文构造在数据库更新之前完成。更新语句本身是单条原子写入，因此失败不会留下部分新记录。

## Windows 运维流程

1. 创建新的外部主密钥文件，配置仅向 VNTS 服务账户授予读取权限。
2. 停止服务并等待状态变为 `STOPPED`：

   ```powershell
   sc.exe stop vnts2
   sc.exe query vnts2
   ```

3. 执行轮换命令。若服务仍在运行，会返回 `Database is already in use by another VNTS process`。
4. 命令成功后，将配置中的 `wireguard_master_key_file` 改为新文件路径。
5. 启动服务并确认日志中的 WireGuard 公钥与轮换前一致：

   ```powershell
   sc.exe start vnts2
   ```

6. 完成服务和连接验证前保留旧主密钥文件；验证完成后按部署方密钥销毁策略处理。

## 为什么不自动改写配置

配置可能包含人工注释、部署模板或并行管理内容。轮换命令只提交数据库密文，不重写 TOML，避免序列化丢失注释或修改无关字段。

数据库轮换成功但配置仍指向旧主密钥时，服务会因 AEAD 认证失败而拒绝启动，不会静默创建新身份。该失败是安全设计的一部分。

## 回滚

若数据库轮换已成功但使用新主密钥启动验证失败：

1. 保持服务停止。
2. 临时将配置的 `wireguard_master_key_file` 指向新主密钥，使轮换命令能认证当前数据库记录。
3. 以旧主密钥文件作为新的命令参数再次执行轮换。
4. 成功后把配置恢复为旧主密钥路径并启动验证。

回滚也是一次新版本的原子重加密，版本继续递增，不会倒退或复用旧 nonce。

## 失败语义

以下情况均返回非零退出码且不修改现有身份记录：

- 服务或另一 VNTS 进程仍持有数据库锁；
- 未提供 `--conf`、数据库不存在、持久化关闭或未配置当前主密钥；
- 当前或新主密钥文件缺失、不可读、长度错误；
- 当前主密钥错误、密文被篡改、公私钥不匹配；
- 新旧主密钥内容相同；
- 主密钥版本溢出；
- 数据库记录在读取后发生变化；
- SQLite 原子更新失败。

错误和日志不输出主密钥、明文私钥或密文内容。

## 验证覆盖

- CLI 强制显式 `--conf` 并正确解析新主密钥路径；
- 第二个 VNTS 进程无法取得数据库锁；
- 成功轮换保持 X25519 公钥不变、版本递增、旧主密钥失效、新主密钥可加载；
- 错误旧主密钥和相同新密钥不改变旧密文；
- 比较并交换拒绝陈旧记录覆盖；
- 真实子进程测试覆盖在线拒绝、停服轮换、旧配置失败关闭、新配置恢复和公钥稳定。

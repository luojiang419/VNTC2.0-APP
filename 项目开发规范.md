# 项目开发规范 / Project Development Guidelines

> 全程使用中文交流。 / Communicate in Chinese throughout.
> 本规范包含编码原则与项目流程管理两大部分。 / This document covers both coding principles and project workflow management.

---

## 第一部分：编码原则 / Part 1: Coding Principles

### 1. 先思考，再编码 / Think Before Coding

**不假设、不隐藏疑惑、暴露权衡点。**
**Don't assume. Don't hide confusion. Surface tradeoffs.**

- 动手前先明确陈述假设，不确定则提问。 / State your assumptions explicitly before implementing. If uncertain, ask.
- 存在多种理解时，列出所有可能，不擅自选其一。 / If multiple interpretations exist, present them — don't pick silently.
- 有更简单的方案就说出来，必要时提出反对意见。 / If a simpler approach exists, say so. Push back when warranted.
- 遇到不清楚的地方立刻停下来，指出困惑点并提问。 / If something is unclear, stop. Name what's confusing. Ask.

### 2. 简洁优先 / Simplicity First

**只写解决问题所需的最少代码，不做任何推测性设计。**
**Minimum code that solves the problem. Nothing speculative.**

- 不添加超出需求的功能。 / No features beyond what was asked.
- 不为一次性代码创建抽象层。 / No abstractions for single-use code.
- 不添加未经请求的"灵活性"或"可配置性"。 / No "flexibility" or "configurability" that wasn't requested.
- 不为不可能发生的场景写错误处理。 / No error handling for impossible scenarios.
- 如果 50 行能搞定却写了 200 行，重写。 / If you write 200 lines and it could be 50, rewrite.
- 自问："高级工程师会觉得这段代码过度设计了吗？" 如果是，就简化。 / Ask yourself: "Would a senior engineer say this is overcomplicated?" If yes, simplify.

### 3. 精准修改 / Surgical Changes

**只触碰必须改的地方，只清理自己造成的遗留。**
**Touch only what you must. Clean up only your own mess.**

- 不"优化"相邻的代码、注释或格式。 / Don't "improve" adjacent code, comments, or formatting.
- 不重构没有问题的东西。 / Don't refactor things that aren't broken.
- 遵循既有代码风格，即使自己不会这样写。 / Match existing style, even if you'd do it differently.
- 发现无关的死代码时，只提及，不删除。 / If you notice unrelated dead code, mention it — don't delete it.
- 移除因本次修改而变得无用的 import、变量、函数。 / Remove imports/variables/functions that YOUR changes made unused.
- 不移除之前就存在的死代码，但是要提醒用户，按用户要求删除与保留。 / Don't remove pre-existing dead code unless asked.
- **检验标准：每一行改动都应能直接追溯到用户的需求。 / The test: Every changed line should trace directly to the user's request.**

### 4. 目标驱动 / Goal-Driven Execution

**定义成功标准，循环验证直到达标。**
**Define success criteria. Loop until verified.**

将任务转化为可验证的目标 / Transform tasks into verifiable goals:

- "添加验证" → "为无效输入写测试，然后让测试通过" / "Add validation" → "Write tests for invalid inputs, then make them pass"
- "修复 Bug" → "写一个能复现该 Bug 的测试，然后让测试通过" / "Fix the bug" → "Write a test that reproduces it, then make it pass"
- "重构 X" → "确保重构前后测试均通过" / "Refactor X" → "Ensure tests pass before and after"

多步骤任务需陈述简要计划 / For multi-step tasks, state a brief plan:

```
1. [步骤/Step] → 验证/verify: [检查项/check]
2. [步骤/Step] → 验证/verify: [检查项/check]
3. [步骤/Step] → 验证/verify: [检查项/check]
```

明确的成功标准可独立循环推进，模糊的标准（"让它能用"）则需要反复确认。 / Strong success criteria let you loop independently. Weak criteria ("make it work") require constant clarification.

### 5. 安全红线 / Security Hard Rules

**绝不在代码中引入 OWASP Top 10 漏洞。**
**Never introduce OWASP Top 10 vulnerabilities in code.**

- 不在 SQL 查询中拼接用户输入，使用参数化查询。 / Never concatenate user input into SQL queries — use parameterized queries.
- 不将用户输入直接插入 HTML，必须转义。 / Never insert user input into HTML unescaped.
- 不在代码中硬编码密钥、密码、Token。 / Never hardcode secrets, passwords, or tokens in source code.
- 不忽略文件路径输入的校验，防范路径遍历。 / Never trust file path inputs without validation — prevent path traversal.
- 发现自己写了不安全的代码，立即修复。 / If you notice you wrote insecure code, fix it immediately.

### 6. 测试策略 / Testing Strategy

**核心路径必须有测试，边缘场景按需补充。**
**Critical paths must have tests. Edge cases are added as needed.**

- 新增功能必须附带至少一个基本功能测试。 / New features must include at least one basic functional test.
- Bug 修复必须附带复现该 Bug 的回归测试。 / Bug fixes must include a regression test that reproduces the bug.
- 重构前后必须确保既有测试全部通过。 / Before and after refactoring, ensure all existing tests pass.
- 不为不可能发生的场景写测试。 / Don't write tests for impossible scenarios.

## 第二部分：项目流程管理 / Part 2: Project Workflow Management

### 7. 任务规划 / Task Planning

- 先做整体规划，不直接开始写完整内容。 / Plan first, don't start writing full content right away.
- 规划文件存放在 `项目开发计划` 文件夹中。 / Store planning files in the `项目开发计划` folder.
- 输出清晰内容：任务目标、拆分步骤、文件/模块清单、待办清单、验收标准。 / Output clearly: task objectives, breakdown steps, file/module list, TODO list, acceptance criteria.
- 每次只完成一小部分，完成后告知当前进度，不一次性输出超长内容。 / Complete one small part at a time, report progress after each, don't output excessively long content at once.

### 8. 进度快照 / Progress Snapshots

每完成一个功能修改，自动生成一份进度快照到 `\进度快照` 文件夹。 / After completing each feature modification, auto-generate a progress snapshot to the `\进度快照` folder.

**快照文件命名 / Snapshot file naming:** `NNN_简要描述.md`（如 `001_用户登录功能.md`） / (e.g., `001_user_login_feature.md`)

**快照模板 / Snapshot template:**

```markdown
# NNN_简要描述 / Brief Description

## 已完成 / Completed
- ...

## 当前修改位置 / Current Edit Position
- 文件/File: `path/to/file`
- 行号/Line: xxx
- 模块/Module: xxx

## 待办清单 / TODO
- [ ] ...
- [ ] ...

## 下一步 / Next Steps
- ...
```

- 若之前的快照缺少序号，在开始修改前将所有快照按时间顺序补充序号。 / If prior snapshots lack numbering, renumber them chronologically before starting the modification.
- 当上下文快满时，主动总结，不要让任务中断。 / When context is near full, proactively summarize — don't let the task break.

### 9. 续写与中断恢复 / Continuation & Recovery

**核心原则：接着上次的进度继续，不重复、不从头来。**
**Core principle: Continue from where the last session ended — don't repeat, don't start over.**

- 记住约定的：格式、风格、结构、命名规范、技术栈。 / Remember agreed conventions: format, style, structure, naming, tech stack.
- 不重复已完成的内容，只从上一次结束的位置继续。 / Don't repeat completed content, continue from where the last session ended.
- 修改时只输出改动部分或完整新版，提前说明是哪种。 / Output only the changed parts or the complete new version — specify which upfront.
- 新开对话、刷新页面、更换设备时，用户提供上一轮的进度快照，直接从上次结束位置继续，不再重复提问确认。 / When starting a new conversation, refreshing, or switching devices, the user provides the previous snapshot — continue directly, no repeated questions.
- 每次操作开始前，先读取之前的进度快照，从上次记录的结束位置分析。 / Before each operation, read the previous snapshots and analyze from the last recorded position.

### 10. 版本控制 / Version Control

**使用 Git 管理代码变更，不依赖手动备份。**
**Use Git for code changes, don't rely on manual backups.**

- 每次修改前确认工作区状态（`git status`）。 / Check working tree status before each modification.
- 完成一个完整功能点后及时 commit，commit message 使用简洁中文描述。 / Commit after completing a coherent feature; use concise Chinese commit messages.
- commit message 格式：`<类型>: <描述>`，类型包括：feat / fix / refactor / docs / style / test / chore。 / Format: `<type>: <description>`. Types: feat / fix / refactor / docs / style / test / chore.
- 如需备份到 `\backup` 文件夹，从最新备份版本号递增。 / If backing up to `\backup`, increment version number from the latest backup.
